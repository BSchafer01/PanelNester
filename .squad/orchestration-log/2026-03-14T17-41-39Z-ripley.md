## Mandate

Phase 3 narrowed to versioned JSON `.pnest` project save/open flows with metadata editing and material snapshots. Parallel ownership split across Parker, Dallas, Bishop, and Hicks.

## Architecture Decision

- **Format:** JSON file-based (`.pnest` extension)
- **Schema Version:** 1 (forward compatible)
- **Persistence Strategy:** Project snapshots materials at save time; import validation against snapshots for existing projects, live library for new imports
- **Scope:** Project CRUD, metadata editing, import/nesting state persistence, bridge extension
- **Deferred:** XLSX import, inline row editing, project export, revision history, multi-material

## Success Criteria

Phase 3 complete when:
1. User can create new project via File menu
2. User can edit all PRD metadata fields
3. User can save project to `.pnest` file via native dialog
4. User can open existing `.pnest` file
5. Imported parts and nesting results persist
6. Materials snapshotted at save time
7. Opening project shows snapshotted material even if library changed
8. All Phase 2 tests still pass
9. New project tests pass
10. Smoke guide updated and verified

## Contracts Extended

**Bridge messages (request/response):**
- `new-project` → `new-project-response`
- `open-project` → `open-project-response`
- `save-project` → `save-project-response`
- `save-project-as` → `save-project-as-response`
- `get-project-metadata` → `get-project-metadata-response`
- `update-project-metadata` → `update-project-metadata-response`

**Error codes:**
- `project-not-found`, `project-corrupt`, `project-unsupported-version`, `project-save-failed`

## Implications

- Project file format stable for v1; version field enables future migrations
- Material snapshots prevent silent corruption from library edits
- Import validation behavior changes: existing projects use snapshots, new imports use live library
- No breaking changes to Phase 2 contracts (additive only)
